2 Project Brief

2.1 Smart Watch Application Description

Our smart watch application is a restaurant locator, making use the google maps API. We envisage that it will have the following features:

Phase 1: before eating

  • Find - Will filter nearby restaurants using a drop-down menu to select the type of food the user wants (eg. French, Korean) with the ability to sort the results by distance or ratings, accessible through another drop-down menu. Note: we came to the conclusion that drop-downs are the best method for this smart watch application as there is limited room for an on-screen keyboard and voice commands aren't yet reliable (or indeed always socially acceptable!).
  • History - Shows the history of restaurants from previous searches so that you can quickly find them again if you don't found anything better in subsequent searches.
  • Favourites - Can selectively add restaurants to a list to quickly look them up again later.
  • Recommendations - A separate function from search, accessed through the main menu of the app that gives you ideas of restaurants that have a good rating. Useful if you don't have a particular type of food that you want.

Selecting a restaurant will bring it up in fullscreen view, with the user able to swipe to the next restaurant. The user should also have the restaurant's contact details shown if available.

Phase 2: On the way

  • Navigation - The search results will have numbered pins on google maps for the restaurant matching the search parameters. The app will have a top down or 45-degree view of google maps which has the yellow line marker showing the path towards the chosen restaurant.

Phase 3: After Eating

  • Like - 5-star system that rates the restaurant you ate at.
  • Share - Recommend to others using facebook or twitter.

2.2 Relationship between emulator and prototype

The emulator will handle all the draw calls, such as the framing of google maps/app GUI within the confines of the visual representation of the smart watch in the browser. It should handle all the API requests that the application needs when prompted by the user through the app GUI. The emulator should be functional in any web browser.

The prototype should rely on the emulator for allowed functionalities of the smartwatch. The prototype is just the GUI(buttons/layout) that the user interacts with to get the location of a restaurant.

2.3 Related products

Restaurant related apps seem to be a lot less prominent in the smart watch market than in the smart phone market. While there are hundreds of apps for smart phones that locate restaurants, there are only a handful of apps that are compatible with smart watches.

  • Eat24 - Is a prominent and consistently recommended app for smart watches and has the following features:
    • Filters - Sort by distance/newest/top rated/delivery & pickup/health.
    • Pay options - Stored credit card info or PayPal/Android Pay.
    • Food-specific searches - Search particulars such as burgers/pizza etc.
    • Favourites.
    • Ability to order from a restaurant directly from the app - This feature is fairly unique to this app, requiring participation/cooperation from the restaurants.
  • Embedded in apps such as TripAdvisor.
  • Franchise specific apps such as PizzaHut.

All restaurant related apps seem to have the basic similarities such as:

  • A method to input the search the type of food they are looking for.
  • Filtering system - Filter based on rating/distance.
  • A rating system - Usually a 5-star system.

Areas they differ in can be:

  • Prices for items on the menu of restaurants.
  • Ability to order at specific restaurants from the app.
  • The output format of the search. Some apps list the restaurants in a scrollable list. Others use pins on a map.

3 Version Control

We will use Git as our version control system. And we will use Github to maintain the service.

  • Working offline - We are not full-time developers with dedicated offices, so we need to be able to work in various different situations. Having a network connection should not affect the workflow. With Git, we can work without the internet, make a commit, browse complete history, merge or create branches, and only push these changes to github once we do get back online.
  • Tolerance for errors - This comes from Git being a distributed version control system. It maintains the full version history on your own disk, with every member of the team able to have a full history of the whole project. It is therefore very hard to break a repository.
  • Workflow - Working with branches is very easy with Git. Combined with its distributed nature, it is possible to design efficient workflows to fit the team's needs.

4 Programming Language

Chosen Language: TypeScript

Reasons:

  • Strongly Typed - fewer bugs and less debugging time.
  • Implements lots of ECMA6 functionalities.
  • Open Source - Biggest maintainer is Microsoft, but it is open source. Google's famous AngularJS uses TypeScript as the official language in its V2 edition.
  • Superset of ECMA(standard of JavaScript). Shares the same syntax system with JavaScript.
  • Well supported by nearly every IDE, especially Intellisense.
  • Class-based, object-orientated design so similar to java in that sense.
  • Support for modules.

Possible cons are mostly implicit:

  • Type annotations - may be cumbersome.
  • Not all third-party libraries have ambient definitions - may have to maintain our own definitions if we want to use a particular library.
  • A compile step is required to transform TypeScript into JavaScript

5 IDE

Integrated Development Environments (IDEs)

License

Licensing is taken into consideration when picking an IDE. If this were a real world project the budgeting and the availability/support of the IDE may factor into which one is needed. Most of the open-source IDEs allow for integration of plugins, adding additional features that are appealing for certain use cases, whereas licensed IDEs such as Webstorm have integrated a lot of features already so there may not be a need for plugins.

Operating System

Most IDEs can be used on Windows, Linux and OSX (Mac) which are big 3 desktop operating systems, and known by most people, although others do exist. Even mobile operation systems such as Android and iOS have IDEs available, though it's doubtful they're used for much serious development. For this project however the main point is to select an IDE can run on Windows, Linux and OSX, as development will be focused on these platforms (with future endeavours of porting to a smart device)

Features

As stated above, most of the open-source IDEs have plugins available that can be incorporated as needed. Taking this into consideration, if there is a feature you want then there is most likely a plugin for it (and if not then there is an opening for a plugin to be developed). With Webstorm it is a licensed program so must be purchased BUT it does have many integrated features so there may not be a need for many plugins. In terms of time this could be an advantage, as although plugins can easily be searched for, it may take time to integrate and learn how to use the features it brings that are needed for a project. On the other hand, if the features needed are already present, time is saved, and focus can instead be on learning how to better use the program.

Open-source IDE

As the open-source IDEs can have many different plugins and offer many of the same features, it is good to get a few reviews from those that have used them. Searching for comparisons of IDEs results in many articles and reviews for a variety of IDEs. Many of them break it down into bullet points, naming features, usability and the general look of the user interface of each IDE. One such article by Craig Buckler compared the IDEs; Atom, Brackets, Light Table and Sublime Text. To quote the article he suggested trying to find something that meets the following criteria:

  1. Suited to web development
  2. Cross-platform and work on Windows, Mac and Linux. Your preferences and settings should be available regardless of which OS you’re using.
  3. General-purpose text editors which support typical web languages. You should be able to use the same application for HTML, CSS, JavaScript, PHP, Ruby, SQL, markdown and more.
  4. Highly customizable with cross-platform plugins and themes.
  5. Fast and stable. Launching should never be a once-a-day dread like some IDEs.
  6. Immediately usable without a steep learning curve or having to remember numerous keyboard shortcuts.

In using this criteria, Craig explained the strengths and weaknesses of each IDE and squared them off against each other, giving each category a winner.

  1. The Interface: Brackets
  2. Syntax Support: Sublime Text
  3. Ease of Use and Learning Curve: Brackets
  4. Speed and Stability: Sublime Text
  5. Native Feature Set: Sublime Text
  6. Web Specific Features: Brackets
  7. Plugins and Extensions: Sublime Text
  8. Customization and Hackability: Atom
  9. The Future: Atom

Although Sublime Text toke most of the categories, Brackets was a close runner up in a selection of this categories.

Alongside individual reviews there are dedicated websites which provide comparisons to help with selecting the correct IDE.

Webstorm

As Webstorm is a licensed program and is available through Otago University resources it can be up for consideration, but if this were a real world project it may be a factor of "do we really need the Webstorm features?" as commercial use requires a license (if there is an intention to publish this application). Due to this Webstorm may not be needed for this project and an open-source IDE could surface with selecting features that are required to complete the project.

What is the best choice?

Brackets would be a good IDE to use, due to it being open-source (developers create pluggins for it) and is maintained very well, available on all platforms (Windows, Mac, Linux) the learning curve (taking into account those within the group that have not had much experience with development) and the ability to incorporate features that are needed to help complete the project. Unfortunately, due to the lack of TypeScript support (the supported version is 2 years out of date) the next best option is Sublime Text, as it was highly rated in reviews, has many of the features that needed for the project, and has a very strong and well supported developer community.

6 Debugging Tools

We will predominantly use Chrome's developer tools for debugging our javascript, as this tool set has very much become the industry standard. This means that there is a lot of (up to date) material online explaining how to use it, including a google sponsored Code School course.

Firefox debugging tool Firebug will of course be used if we encounter issues that appear in Firefox but not in chrome, to try and discover what is causing the differing behavior. Between these two most bugs should be found. Firebug is actually in the process of unifying itself with native Firefox developer tools. The Firefox developer edition incorporates this, as well as a number of other tools, and is actually something that we might look into if have the time and inclination further down the line.

Both Chrome's developer tools and firebug support source maps, an important feature as we'll be using TypeScript, so would like to debug using our source rather than the compiled javascript that the developer tools would otherwise see.

In the end it doesn't really matter too much which one we use (Microsoft also have developer tools in their Edge browser, but it is not widely used by developers nor by the general public, so lacks supporting material), as both serve the same purpose, and go about it in only slightly different ways. We can set breakpoints in both, step through the code line by line, query values at different points in the code and so on.

7.3 Some Details

In Design, learn, architecture, test and publish phases, everyone will get involved, the deadline for which are listed above. Things get a little tricky when the development phase starts since the whole project is split into two parts: platform and app. The team will be divided into 2 subteams, one subteam is responsible for the platform, the other is responsible for the smart watch.

Problem:
This is like traditional web programming; front-end group(smart watch) and back-end group(platform). Here we will face the first problem; the development of the platform and smart watch will both get the UI part and logic part. But the smart watch team will rely on the platform because the smart watch sits on top of it, so will need the platform to deliver some API which it can then use to play with.

Solution:
The platform subteam will first develop some fake API for the test case while the smart watch subteam work on the UI. This means that as soon as the smart watch subteam finishes the UI phase, it will immediately receive all the API it needs to implement the functionalities even if the return data is faked, the UI can be tested! The platform subteam will then has sufficient time to implement the real business logic and UI part of the platform.
