2.1 Smart Watch Application Description

Our smart watch application is a restaurant locator using the google maps API that has the following functionalities:

Phase 1: before eating

  • Find - The restaurant using a drop down menu for the type of food you want (eg. French, Korean) and by using a drop down filter for the results such as distance/ratings. Note: we came to the conclusion that drop downs are the best method for this smart watch application as there is limited room for an onscreen keyboard and voice commands aren't reliable.
  • History - Shows the history of previously searched restaurants so that you can quickly find the searched restaurant if you haven't found anything better than it.
  • Favorites - Can selectively add restaurants to a list for quick lookup later.
  • Recommendations - A separate function from search accessed through the main menu of the app that gives you ideas of restaurants that have a good rating if you don't have a particular type of food that you want.

Individually selected restaurants will be shown fully on the screen and the user should be able to swipe to the next restaurant. The user should also have the restaurant's contact details shown if available

Phase 2: On the way

  • Navigation - The search results will have numbered pins on google maps for the restaurant matching the search parameters. The app will have a top down or 45 degree view of google maps which has the yellow line marker showing the path towards the chosen restaurant.

Phase 3: After Eating

  • Like - 5 star system that rates the restaurant you ate at.
  • Share - Recommend to others using facebook or twitter.

2.2 Relationship between emulator and prototype

The emulator will handle all the draw calls such as the framing of google maps/app GUI within the confines of the smartwatch visual representation in the browser. It should handle all the API requests that the application needs when prompted by the user through the app GUI. The emulator should be functional in any web browser.

The prototype should rely on the emulator for allowed functionalities of the smartwatch. The prototype is just the GUI(buttons/layout) that the user interacts with to get the location of a restaurant.

2.3 Related products

Restaurant related apps seem to be a lot less prominent in the smartwatch/android wear market than in the android smartphone/browser market. While there are hundreds of apps for smartphones that locate restaurants, there are only a handful of apps that are compatible with smart watches.

  • Eat24 - Is a prominent and consistently recommended app for smart watches and has the following features:
    • Filters - Sort by distance/newest/toprated/delivery & pickup/health
    • Pay options - Stored credit card info or PayPal/Android Play
    • Food specific searches - Search particulars such as burgers/pizza etc
    • Favourites
    • Ability to order from a restaurant directly from the app - This feature is fairly unique to this app though it requires participation/cooperation from the restaurants
  • Embedded in apps such as TripAdvisor
  • Franchise specific apps such as PizzaHut

All restaurant related apps seem to have the basic similarities such as:

  • A method to input the search the type of food they are looking for
  • Filtering system - Filter based on rating/distance
  • A rating system - Usually 5 star system

Areas they differ in can be

  • Restaurant specific prices for each item on the menu
  • Ability to order at specific restaurants from the app
  • Output format of the search. Some apps list the restaurants in a scrollable list. Others use pins on a map.

3 Version Control

We will use Git as our version control system. And we will use Github to maintain the service.

  • Offline Working - We are not full-time coder, so we should work under various situation. The connection of network should not affect the workflow. With Git, we can work without internet, make a commit, browser complete history, merge or create branches.
  • Tolerance for errors - Since Git is a distributed version control system. It maintains the full version history in your own disk, every member of the team might have full history of the whole project. So, it is very hard to break a repository.
  • Workflow - Working with branch is very easy with git. Combining its distributed essential, you can design a workflow which exactly fits your team best.

7.3 Some Details

In Design, learn, architecture, test and publish phase, everyone will get envolved, and the deadline is listed above. Things get a little tricky when the develop phase start, since the whole project is about two part: platform and app. The whole team will be divided into 2 subteams, one subteam is responsible for the platform, the other is responsible for the smart watch.

Problem:
This is like the traditional web programming, front-end group(smart watch) and back-end group(platform). Here we will face the first problem, the develoment of platform and smart watch will both get the UI part and logic part. But the smart watch team will reply on the platform because the smart watch sits on top of the platform, it needs the platform to deliver some API which it can then use to play with.

Solution:
The platform subteam will first develop some fake API for the test case while the smart watch subteam work on the UI. This means when as soon as the smart watch subteam finish the UI phase, it will immediately receive all the API it needs to implement the functionalities even the return data is faked, they can test the UI with data! And the platform subteam now have sufficient time to implement the real business logic and UI part of platform.
